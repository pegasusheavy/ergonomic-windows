String Conversion Tips

Use WideStringBuilder for multiple strings

When building strings from multiple parts, use WideStringBuilder with pre-allocated capacity: 

// ‚ùå Multiple allocations
let s1 = to_wide("Hello");
let s2 = to_wide(", ");
let s3 = to_wide("World");

// ‚úÖ Single allocation with builder
let wide = WideStringBuilder::with_capacity(100)
    .push("Hello")
    .push(", ")
    .push("World")
    .build();

Reuse builders in loops

Use build_and_clear() to reuse the same builder: 

let mut builder = WideStringBuilder::with_capacity(256);

for name in names {
    builder.push(name);
    let wide = builder.build_and_clear(); // Returns buffer, clears for reuse
    process_name(&wide);
}

Reducing Allocations

Use WideStringPool for high-throughput

For loops that process many strings, use WideStringPool to eliminate allocation overhead entirely: 

// Pre-allocate a pool of buffers
let mut pool = WideStringPool::with_preallocated(16, 256);

for filename in files {
    let pooled = pool.get(filename); // Zero allocations!
    
    // Use pooled.as_pcwstr() with Windows APIs
    some_api(pooled.as_pcwstr());
    
    pool.put(pooled); // Return buffer for reuse
}

Keep WideStrings alive

If you're making multiple calls with the same string, convert it once: 

// ‚ùå Allocates on every call
for _ in 0..100 {
    api_call(WideString::new("constant").as_pcwstr());
}

// ‚úÖ Allocate once
let constant = WideString::new("constant");
for _ in 0..100 {
    api_call(constant.as_pcwstr());
}

Use with_capacity

When you know the approximate size upfront: 

// Pre-allocate for known-size data
let ws = WideString::with_capacity(MAX_PATH);

let builder = WideStringBuilder::with_capacity(estimated_len);

Profiling Your Code

Run benchmarks

# Run all benchmarks
cargo bench

# Run specific benchmark
cargo bench --bench string_bench

# Run allocation tracking
cargo bench --bench allocations

Memory profiling with dhat

# Run memory profiler
cargo run --bin memory-profile --features dhat-heap

# Open dhat-heap.json in the DHAT viewer:
# https://nnethercote.github.io/dh_view/dh_view.html

CPU profiling with flamegraph

# Install cargo-flamegraph
cargo install flamegraph

# Generate flamegraph (requires Administrator on Windows)
cargo flamegraph --bin cpu-profile

Fuzz testing

The crate includes fuzz tests to find edge cases and potential crashes: 

# Install cargo-fuzz (requires nightly)
cargo install cargo-fuzz

# On Windows, install LLVM for ASAN support
winget install LLVM.LLVM

# Run fuzz tests
cd fuzz
cargo +nightly fuzz run fuzz_strings -- -max_total_time=60

‚úì Fuzz tested: All string conversion functions have been fuzz tested with millions of inputs including Unicode edge cases, embedded nulls, and invalid UTF-16.

Measure First

Don't optimize prematurely. The default implementations are already fast. Only optimize after profiling shows a specific bottleneck.
